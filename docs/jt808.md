# JT808 GPS Tracker Protocol

## Overview

JT/T 808-2019 is the Chinese national standard for vehicle GPS tracker communication. Unlike the P901 (SMS-only), JT808 devices maintain **persistent TCP connections** and **push location data autonomously** — no polling required.

**Driver class:** `App\Drivers\Jt808Driver` (Phase 2 — coming)
**Communication mode:** `active` (device pushes to server continuously)
**Server port:** TCP 7018
**Server microservice:** `jt808-server/` (Go binary)

---

## Key Differences from P901 / GF-07

| Feature | P901 / GF-07 | JT808 devices |
|---|---|---|
| Location delivery | SMS reply (passive poll) | TCP push (autonomous) |
| Connection type | None (SMS only) | Persistent TCP |
| Real-time tracking | Configurable intervals | Continuous (configurable) |
| Auth | No authentication | Token-based 2-step auth |
| SOS | SMS to phone number | Alarm flag in location frame |
| Commands | SMS text | TCP binary frames |
| Heartbeat | N/A | 0x0002 every 60–120 s |
| Batch uploads | N/A | 0x0704 (offline buffering) |

---

## Protocol Frame Format

```
0x7e | MsgID(2) | Attrs(2) | Phone-BCD(6) | SeqNum(2) | [SubpkgHdr(4)] | Body(N) | Checksum(1) | 0x7e
```

- **Escape rules**: `0x7e` in payload → `0x7d 0x02`; `0x7d` → `0x7d 0x01`
- **Checksum**: XOR of all bytes from MsgID through Body
- **Phone**: 12-digit BCD (unique device identifier — maps to IMEI or SIM)

## Device Authentication Lifecycle

```
Device connects (TCP)
  │
  ├─ 0x0100 Registration  ──► Server stores phone, issues auth token
  │   └─ 0x8100 Response + token
  │
  ├─ 0x0102 Authentication + token ──► Server validates token in Redis
  │   └─ 0x8001 ACK (result=0: success)
  │
  └─ [now authenticated]
      ├─ 0x0200 Location reports (continuous)
      ├─ 0x0002 Heartbeat (every 60–120 s)
      └─ 0x0704 Batch location (if device was offline)
```

## Key Message IDs

### Terminal → Platform

| ID | Name | Description |
|---|---|---|
| 0x0001 | Terminal Response | ACK to server commands |
| 0x0002 | Heartbeat | Keepalive (empty body) |
| 0x0100 | Registration | Device startup; gets auth token |
| 0x0102 | Authentication | Presents token to complete auth |
| 0x0200 | Location Report | Single GPS fix with status/alarm flags |
| 0x0704 | Batch Location | Multiple fixes (used after offline period) |

### Platform → Terminal

| ID | Name | Description |
|---|---|---|
| 0x8001 | Platform Response | ACK to terminal messages |
| 0x8100 | Registration Response | Result + auth token |
| 0x8103 | Set Terminal Params | Configure reporting interval, etc. |
| 0x8201 | Query Location | Force immediate location report |

## Location Report (0x0200) Body

| Offset | Bytes | Field | Unit |
|---|---|---|---|
| 0 | 4 | Alarm Flags | bit field |
| 4 | 4 | Status Flags | bit field |
| 8 | 4 | Latitude | 10⁻⁶ degrees |
| 12 | 4 | Longitude | 10⁻⁶ degrees |
| 16 | 2 | Altitude | metres |
| 18 | 2 | Speed | 0.1 km/h |
| 20 | 2 | Direction | degrees (0=north) |
| 22 | 6 | Timestamp | BCD YYMMDDHHmmss |
| 28 | N | Additional Items | TLV |

### Key Alarm Bits

| Bit | Meaning | Platform Event |
|---|---|---|
| 0 | SOS | `SosPressed` → Critical Incident |
| 1 | Overspeed | `Overspeed` → High Incident |
| 7 | Low battery | `LowBattery` → Medium Incident |
| 11 | Power low | `PowerLow` → Medium Incident |
| 12 | Power off | `PowerOff` → High Incident |

### Key Status Bits

| Bit | Meaning |
|---|---|
| 0 | ACC on (ignition) |
| 1 | Located (GPS fix valid) |
| 2 | Latitude south (0=north) |
| 3 | Longitude west (0=east) |
| 4 | Stopped (0=running) |

## ACK Discipline

**ALWAYS ACK every message the device sends.**

- ACK within 2 seconds. Devices retransmit after 5 attempts if no ACK.
- ACK the location frame BEFORE publishing to Redis.
- For SOS: ACK immediately, escalate asynchronously.
- For unknown message IDs: ACK with result=3 (unsupported) so the device stops retrying.

---

## Heartbeat Timeout

- Devices send 0x0002 every 60–120 seconds (configurable via 0x8103).
- Server sets TCP read deadline to 3 minutes. If no frame arrives:
  - Connection is closed.
  - Session removed from Redis online ZSET.
  - Laravel publishes a `DeviceOffline` event.
- OS-level TCP keepalive is also enabled (60 s) to detect dead peers faster.

## Horizontal Scaling Architecture

```
Devices (TCP)
     │
     ▼
HAProxy (source-IP hash, TCP mode)
     │
     ├─ JT808 Node 1 (host port 7018) ─── Redis ─── Laravel Workers
     ├─ JT808 Node 2 (host port 7018) ─/           └─ Soketi → React
     └─ JT808 Node 3 (host port 7018) ─/
```

**Why source-IP hashing**: A device's GPRS IP may change on reconnect, but usually stays the same within a session. Hashing routes it to the same node, minimising re-authentication overhead. Redis still holds all auth tokens, so any node can authenticate any device.

**Command routing (Laravel → Device)**:
Laravel publishes to Redis pub/sub channel `jt808:cmd:{phone}`. All JT808 server replicas subscribe. The replica that holds the device's TCP connection forwards the command. Replicas that don't have the device ignore it.

## Database Architecture

### MySQL (operational data)
- `device_locations` — latest location per device (upserted by phone)
- `device_location_history` — time-partitioned history (prune > 90 days)
- `incidents` — generated from alarm flags

### InfluxDB (time-series, Phase 2)
- Measurement: `device_gps`
- Tags: `phone`, `device_type=jt808`
- Fields: `lat`, `lon`, `speed`, `altitude`, `direction`, `gps_fixed`, `acc_on`
- Retention: 1 year (raw) → 5 years (downsampled to 1-minute intervals)

### Trip detection (future)
- Trip starts when ACC on + movement detected for > 60 s
- Trip ends when stopped + ACC off for > 5 minutes
- Stored in `trips` table with polyline geometry

## Security Considerations

1. **Auth token per device** — validated in Redis on every reconnect.
2. **Auth timeout** — unauthenticated connections closed after 30 s.
3. **Message rate limiting** — add Redis INCR rate limiter if needed.
4. **Firewall** — port 7018 should only be reachable from device GSM IPs (if carrier provides a range).
5. **No TLS** — JT808 devices rarely support TLS. Consider a VPN (WireGuard) or carrier APN restriction.
6. **Input validation** — all binary fields validated before use; `parseIncoming` never panics.
